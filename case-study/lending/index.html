Deep Problem Framing
The Goal

The platform only supported equity investments. Clients were asking for private lending capabilities — mortgage funds, bridge loans, real estate debt. We needed to extend the platform to handle debt instruments without building a separate product or confusing existing equity users.

The Problem

Debt UX is fundamentally different from equity UX. Equity users think in shares, ownership percentages, and exit events. Debt users think in repayment schedules, interest rates, and default risk. The existing platform had no concept of loan creation, repayment tracking, interest accrual, or default escalation. Bolting lending onto an equity platform without careful design would confuse both user bases.

How Might We

How might we extend an equity-focused platform to support private lending — without creating a separate product or confusing existing equity users?

Before Equity only Platform couldn't handle debt instruments, repayment schedules, interest, or defaults.
After Equity + Debt Full lending workflows — loan creation, repayment schedules, interest accrual, default escalation.

User Research
Methodology

Stakeholder interviews with mortgage fund managers, bridge loan operators, and real estate debt specialists. Workflow observation sessions with existing equity users to understand mental model differences. Competitive analysis of lending-specific platforms.

Sample

8 mortgage fund managers, 5 bridge loan operators, 10+ existing equity users. Research conducted across 3 months before design began.

Key Insights
Debt users think in schedules, not shares

Equity users care about ownership percentage and valuation. Debt users care about when they get paid, how much interest accrues, and what happens if a borrower defaults. The mental models are fundamentally different.

Default workflows can't be an afterthought

In equity, a failed investment is a write-off. In lending, a default triggers a complex escalation — notices, cure periods, legal action. Default handling had to be designed as a first-class workflow, not an edge case.

One platform, two product personalities

Users needed the platform to feel purpose-built for their asset class. An equity manager opening the platform should see equity-relevant information. A lending manager should see loan-relevant information. Same platform, contextually different experience.

Design Approach / Solution

Our team designed the private lending module — from loan creation through repayment tracking and default escalation. The core challenge was building a contextual UI that adapted based on whether the user was managing equity or debt, without fracturing the product into two separate applications.

The solution used a shared infrastructure layer (users, documents, compliance) with asset-class-specific interfaces on top. Equity users never saw lending UI, and lending users never saw equity UI — but administrators could manage both from a single dashboard.

Key Features

1. Loan creation wizard with term sheets and interest configuration

2. Repayment schedule visualization with payment tracking

3. Interest accrual dashboard with real-time calculations

4. Default escalation flows with notices and cure periods

5. Trulioo and Onfido identity verification integration

6. TFSA/RRSP registered account support

7. Auto-invest reinvestment rules

8. Contextual UI that adapts for debt vs. equity users

Design Decisions
01

Contextual UI over separate applications

Considered

Separate lending app, tabbed interface, contextual adaptation

Chose & Why

Contextual adaptation — building a separate app would double maintenance and confuse clients who manage both equity and debt. The contextual approach let administrators see everything while individual users only saw their asset class.

Equity vs. Debt Context Switching
02

Visual repayment schedules over table-only views

Considered

Spreadsheet-style tables, Gantt-style timeline, visual schedule with table fallback

Chose & Why

Visual schedule with table fallback — fund managers needed to see the full repayment timeline at a glance, not scroll through rows. The visual timeline showed payment status (paid, upcoming, overdue, defaulted) with color coding, while the table view remained available for exports and detailed analysis.

Reflection
✓ What Worked

1. Contextual UI kept both user bases happy — equity users never saw lending complexity, lending users got a purpose-built experience

2. Shared infrastructure reduced development time — compliance, documents, and user management worked identically across asset classes

3. Default escalation as a first-class workflow prevented edge-case disasters — fund managers trusted the system to handle the hardest scenarios

◈ What I'd Do Differently

1. Earlier user testing with actual lending clients — we designed based on equity-user assumptions that didn't always transfer

2. More sophisticated interest calculation UI — the first version was too simplified for complex amortization schedules

3. Mobile experience for borrowers — the initial release was desktop-only, and borrowers increasingly expected mobile access
