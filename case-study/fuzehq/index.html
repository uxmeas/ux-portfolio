Deep Problem Framing
The Goal

Teams using AI agents have no control plane. Agents run in terminals with no visibility, no skill management, and no coordination. I set out to build a single workspace where humans and AI agents organize, execute, and track work together.

The Problem

Before FuzeHQ, managing AI agents required juggling 5 separate tools — a project manager, an AI interface, documentation, deployment, and team chat. Status updates were wrong 40% of the time because agents had no structured way to report progress. There was zero visibility into what agents were working on, what skills they had, or how they coordinated.

How Might We

How might we create a single workspace where humans and AI agents organize, execute, and track work together — eliminating context switching and providing full visibility into agent work?

Before 5 tools · 40% wrong status PM + AI + docs + deploy + chat. No visibility into agent work. Constant context switching.
After 0 context switches 9 agents, 14 MCP tools, work and tracking unified in one platform.
User Research
Methodology

Dogfooding — used FuzeHQ daily as our own team management platform. Observed how 2 human co-founders coordinated 9 AI agents across multiple projects simultaneously.

Sample

2 co-founders as primary users, 9 AI agents as managed entities. Multiple projects running in parallel over months of daily usage.

Key Insights
Agents need skills, not just prompts

Generic prompts led to inconsistent output. Structured skills with governance, versioning, and progressive loading transformed agent reliability and made expertise reusable across tasks.

Context switching kills productivity

Jumping between PM tool, AI interface, docs, and deployment meant losing context every time. Unifying work and tracking in one platform eliminated the fragmentation entirely.

No visibility into agent work

Without a control plane, agents were black boxes. You couldn't see what they were doing, whether they finished, or if their work was correct. QA gates and status tracking made agent work auditable.

Design Approach / Solution

I designed FuzeHQ as a control plane for human-AI teams — where organizing, executing, and tracking happen in one workspace. The platform treats AI agents as first-class team members with roles, skills, budgets, and accountability.

The key architecture decision was MCP (Model Context Protocol) execution over embedded AI. Agents run locally via CLI but report through the platform, keeping compute costs on the user's subscription while centralizing visibility and governance.

Key Features

1. Agent roster with skills, budgets, and role assignments

2. Task board with agent assignment and QA gates

3. MCP server with 14 tools for agent coordination

4. Cross-machine collaboration via structured comments

5. Mandatory QA gates — no self-certification allowed

Design Decisions
01

MCP execution over embedded AI

Considered

Embedded AI in the platform, API-based agents, MCP local execution

Chose & Why

MCP local execution — agents run on the user's machine using their existing Claude subscription. The platform provides coordination, not compute. This keeps costs predictable and lets agents access local files, repos, and tools directly.

MCP Architecture Diagram
02

No self-certification — mandatory QA gates

Considered

Agent self-reporting, peer review, mandatory QA agent

Chose & Why

Mandatory QA gates — after an incident where 28 features shipped unverified because agents marked their own work as "done," we designed a system where builders can only set status to "in review." Only the QA agent can move tasks to "done" after posting a QA pass.

Reflection
✓ What Worked

1. Dogfooding — using FuzeHQ daily to manage our own AI team meant every pain point was felt immediately and fixed quickly

2. MCP architecture — local execution with centralized coordination hit the sweet spot between control and cost

3. Named agents with skills — treating agents as team members with expertise, not generic chatbots, transformed how work got done

◈ What I'd Do Differently

1. Start with the dashboard, not settings — we built agent configuration before building the visibility layer that users actually need first

2. Design for multi-tenant from day one — retrofitting tenant isolation was painful and introduced security debt

3. Surface-layer polish must match ambition — the product vision was strong but early UI didn't reflect it
