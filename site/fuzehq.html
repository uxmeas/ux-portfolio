( 01 ) The Problem

Status updates were wrong 40% of the time because agents had no structured way to report progress. Teams using AI agents have no control plane. Agents run in terminals with no visibility, no skill management, no coordination. The standard setup required 5 separate tools: a project manager, an AI interface, documentation, deployment, and team chat. There was no way to see what an agent was working on, whether its output had been verified, or how it connected to the broader project. The problem wasn't that AI agents weren't capable. It was that every management layer around them was missing.

( 02 ) How It Was Framed

The research methodology was dogfooding. FuzeHQ was used daily as the actual management system for 2 co-founders running 9 AI agents across multiple simultaneous projects. Every pain point was felt in real time and addressed in the next build cycle. The insight that drove the core architecture: agents didn't need more AI capability. They needed accountability infrastructure. Skills, budgets, roles, and QA that made their work auditable.

HMW

How might we create a single workspace where humans and AI agents organize, execute, and track work together, with full visibility and zero context switching?

( 03 ) The Answer
A / 01

MCP local execution over embedded AI

Three execution models were evaluated: embedding AI directly in the platform, routing through API-based agents, and MCP local execution. Embedding AI in the platform would have added significant per-usage cost and prevented agents from accessing local files, code repositories, and development tools. API-based routing would have added latency and required managing additional API credentials. MCP local execution was the right architecture. Agents run on the user's machine using their existing Claude subscription, and the platform provides coordination and visibility rather than compute. Costs stay predictable. Agents have full access to local resources. The platform's value is the control layer, not the AI layer.

A / 02

Structured skills replaced generic prompts

The first version used free-form prompts to direct agent behavior. Output was inconsistent. The same agent produced different quality work depending on which session it was operating in. Structured skills with explicit governance, version history, and progressive loading transformed agent reliability. Each skill defined exactly what the agent knew how to do and how it should approach the task. 51 skills across 9 agents. Expertise became persistent, reusable, and improvable rather than reconstructed from scratch in every conversation.

A / 03

Mandatory QA gates made agent work auditable

After an incident where 28 features shipped unverified because agents had self-certified their own work as complete, the status system was redesigned from the ground up. In the new system, agents operating as builders can only move tasks to "in review," never to "done." Only the designated QA agent can close a task after posting a QA pass. Self-certification is architecturally blocked, not just discouraged by policy. The distinction matters. A policy can be overridden by a tired human. An architectural constraint cannot.

( 04 ) Key Screen Annotations
SCREEN 01

Agent Roster

Each agent has a named profile with role, assigned skills, active task count, and budget. The roster is the team page. It makes AI agents legible as collaborators rather than anonymous processes running in a terminal.

SCREEN 02

Task Board with QA Gates

Every task shows its agent assignment, current status, and QA gate state. A task cannot move to "done" without a QA pass visible in the activity log. The board is the truth, not whatever an agent reported in a chat window.

( 05 ) Results

9 agents with distinct roles · 14 MCP tools · 51 structured skills · 0 context switches · Work and tracking unified in one platform · Mandatory QA gates preventing self-certification

( 06 ) Reflection
What worked

Dogfooding created immediate feedback loops. Every frustration became a design problem the same week it was felt. MCP architecture hit the correct balance between control and cost. Naming agents and giving them skills changed the working relationship from "using a chatbot" to "managing a team."

What to do differently

Build the dashboard first, not the settings. The initial sprint focused on agent configuration and skill management before the visibility layer was in place. The team was configuring a system they couldn't yet see working. Design for multi-tenant isolation from the initial architecture. Retrofitting tenant separation introduced security review work that slowed the beta launch significantly.
