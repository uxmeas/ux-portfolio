( 01 ) The Problem

30 to 60 minutes per document. Zero creative output. Every designer converting content from documents into layouts does the same invisible labor. Open the Word doc, read the heading hierarchy, manually create an H1 in Figma, set the style, type the text, create an H2, set that style, add body text. Repeat for every heading, paragraph, list item, and table. The content already has structure. It's there in the document, expressed as heading levels and paragraph styles. But there's no bridge between document formats and design formats, so designers ignore the structure entirely and recreate it by hand. Universal invisible cost that nobody talks about because it's just how it's done.

( 02 ) How It Was Framed

The observation was simple. Every designer does this translation. It's not a niche workflow or a pain point specific to a particular domain. It's structural work that happens before any creative work can begin. Invisible labor that every design team has factored into timelines so thoroughly that nobody questions it anymore.

HMW

How might we go from document to design structure in under 60 seconds, with hierarchy auto-detection that maps document structure directly to design tokens?

( 03 ) The Answer
A / 01

Auto-detection over manual configuration

The plugin reads the document's heading structure: H1, H2, H3, body text, lists, tables. It maps each level to the corresponding design token automatically. Designers don't configure the mapping from scratch on each import. They review what was detected, confirm or adjust style assignments, and generate. The shift from "configure the tool" to "review the output" removes the cognitive burden that makes this kind of translation work exhausting. Correct defaults mean most imports require no adjustments at all.

A / 02

Dual mode served two distinct user types without compromise

Early testing revealed two fundamentally different user approaches. Designers new to the workflow wanted to be walked through every step, understand what was happening, and feel confident before generating output. Power users wanted to skip every explanation and apply their saved settings in under 10 seconds. Building a single interface for both groups would have failed both. The wizard mode walks through 4 structured steps: import, structure review, style mapping, preview, before generating. Compact Pro condenses the entire flow into a single panel. Both modes produce identical output. Neither forces a compromise.

A / 03

Structure-first shipped a useful product faster

The first planning discussion debated whether to build full pixel-perfect content rendering. Preserving not just hierarchy but exact formatting, tables, and complex layouts. This would have taken months and produced a brittle product that broke on edge cases. The decision to focus on hierarchy detection first meant the plugin was useful from day one. Designers got their H1s, H2s, and body text blocks into Figma in the correct hierarchy within 60 seconds. Content formatting could be refined after. The structural skeleton was 80% of the work.

( 04 ) Key Screen Annotations
SCREEN 01

Structure Detection Panel

Shows the detected hierarchy as a tree. Each node labeled with its document role and the Figma design token it will map to. Designers can override any assignment before generating. The panel makes the auto-detection logic visible rather than opaque.

SCREEN 02

Compact Pro Mode

The full import-detect-map-generate flow compressed to a single scrollable panel with inline controls. No steps, no wizards, no explanations. Token mapping saved from previous sessions auto-applies. Import to generation in under 10 seconds for regular users.

( 05 ) Results

60 to 1 min translation time · 98% time reduction · 2 UX modes · Auto-detect hierarchy · Brand identity system for consistent token mapping across teams

( 06 ) Reflection
What worked

Dual-mode UX was the correct call. The wizard captured users who would have bounced from a single-panel interface, while Compact Pro retained power users who would have abandoned a wizard. Structure-first approach meant a useful product launched instead of a perfect product that wasn't ready.

What to do differently

Google Docs and Notion support should have been in the first version. Markdown import was the third most-requested feature in the first week. The plugin validated the concept but the web SaaS version, which reaches non-Figma designers and handles more input formats, should have been built in parallel, not as a follow-on phase.
