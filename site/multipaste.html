( 01 ) The Problem

Figma has no native mechanism to distribute structured data across multiple text layers. Building a table mockup with 50 rows means selecting a layer, pasting, selecting the next, pasting. 50 times. For a single table mockup with real data, that's 15 minutes of repetitive work with zero creative output. The problem isn't that designers can't do it. It's that they do it constantly. Every table, every data-heavy component, every time a content update requires pushing new values into an existing layout. The work is invisible because it has no name, but it adds up to hours across a project.

( 02 ) How It Was Framed

This was built from personal daily pain. The select-layer, paste, next-layer, paste cycle identified in direct workflow observation as the exact friction point. No formal research phase. The brief was simple: eliminate the thing that takes 15 minutes and produces nothing.

HMW

How might we let designers paste structured data into dozens of text layers in one action, eliminating 15 minutes of repetitive work per table?

( 03 ) The Answer
A / 01

Preview before apply, so mistakes never land silently

The first prototype applied data directly to selected layers with no preview. The result was predictable. Ordering mismatches, wrong values in wrong layers, and the need to undo-reorder-retry on every error. A live preview panel was added that shows exactly which text value will go to which layer before anything is applied. In testing, preview caught 100% of ordering mistakes before application. The operational impact: what was previously a stressful batch operation became a confident one-click action.

A / 02

Selection order made explicit and visible

Figma's internal selection order doesn't always match visual order. Layers selected top-to-bottom visually can register in a different sequence internally depending on how they were created. Displaying the selection sequence explicitly in the preview panel meant designers could see the actual paste order before committing. The difference between "shows the mapping" and "trusts the mapping" was the difference between a plugin designers used once and one they used daily.

A / 03

Empty value handling preserved existing content

Real data has gaps. Spreadsheet columns have empty cells. JSON arrays have null values. The first version blindly overwrote every layer, including ones that should have kept their placeholder text. Toggle controls for "skip empty values" and "preserve existing text when empty" were added so the plugin handled real-world data structures rather than idealized ones.

( 04 ) Key Screen Annotations
SCREEN 01

Preview Panel

The left column shows each value from the pasted data. The right column shows the target layer. Designers can reorder before applying. This single interface element is the reason the plugin is trusted with production mockups.

SCREEN 02

Delimiter Controls

Tab, comma, newline, pipe, and custom delimiters are all supported. The plugin auto-detects the most likely delimiter from the pasted content and sets it as the default, reducing the configuration step for most use cases to zero.

( 05 ) Results

15 to 1 min table population · 93% time reduction · 7.2KB bundle, zero dependencies · 12 features shipped · 2 weeks from first line to published plugin

( 06 ) Reflection
What worked

Scratching the itch directly. Built because it was needed, which meant requirements were clear and validation was immediate. Preview-first design eliminated the undo-retry loop that makes batch operations feel risky.

What to do differently

Spreadsheet import. Direct .xlsx or .csv file upload would eliminate the step of opening a spreadsheet, copying, and pasting into the plugin input. Analytics tracking from the start. No telemetry means no data on which delimiter types are most used, whether preview is actually being checked, or where users abandon the flow.
